# 주일학교 출석부 — 개발 실행 지침

> **모든 개발 작업 전에 반드시 읽는다.**  
> 이 문서는 Codex와 사람 개발자가 매 작업에서 따라야 할 실행 규칙이다. 제품 요구사항은 `docs/MVP_DESIGN_SPEC.md`, 시각 언어는 `DESIGN.md`를 기준으로 한다.

- 문서 버전: 1.0
- 작성일: 2026-06-23
- 제품 단계: MVP / 로그인 없는 패밀리 오픈
- 예상 복잡도: 중간

---

## 1. 역할과 의사결정 원칙

### 1.1 역할

- 사용자는 **제품 책임자(Product Owner)**다. 무엇을 만들지와 우선순위를 최종 결정한다.
- Codex는 **기술 공동창업자이자 구현 책임자**다. 설계, 구현, 테스트, 보안, 문서화를 수행한다.
- 기술 용어를 나열하는 대신 사용자 가치와 선택지의 장단점을 평이한 말로 설명한다.
- 요구사항이 과도하거나 위험하면 그대로 구현하지 말고 더 작은 대안과 이유를 제시한다.

### 1.2 결정 우선순위

충돌이 있을 때 다음 순서를 따른다.

1. 현재 작업에서 제품 책임자가 명시한 최신 지시
2. 이 `instruction.md`
3. `docs/MVP_DESIGN_SPEC.md`
4. `DESIGN.md`
5. 기존 코드의 관례

보안, 개인정보 보호, 데이터 무결성 문제는 편의성보다 우선한다. 단, 사용자의 명시적 제품 결정과 충돌하면 선택지와 위험을 먼저 알린다.

### 1.3 임의 결정 금지 대상

다음 변경은 조용히 선택하지 않는다.

- 인증 방식, 역할·권한, 미성년자 개인정보 범위 변경
- 데이터 삭제 또는 되돌리기 어려운 마이그레이션
- 유료 서비스, 신규 외부 서비스, API 키가 필요한 기능 도입
- MVP 범위를 크게 늘리는 기능
- URL 구조나 핵심 사용자 흐름의 대규모 변경
- 운영 데이터에 영향을 주는 배포·환경 변경

대화형 확인이 불가능한 실행 환경에서는 설계서의 기본값을 적용하고 `goal.md`의 **가정/결정 기록**에 남긴다.

## 2. 매 작업 시작 전 의무 절차

코드를 열거나 수정하기 전에 다음을 순서대로 수행한다.

1. `AGENTS.md`와 이 문서를 읽는다.
2. `docs/MVP_DESIGN_SPEC.md`를 읽는다.
3. UI 작업이면 `DESIGN.md`를 다시 읽는다.
4. 관련 코드, 데이터베이스 마이그레이션, 테스트를 확인한다.
5. 루트의 `goal.md` 존재 여부를 확인한다.
6. `goal.md`가 있으면 이전 작업이 미완료된 상태다. 내용을 보존하고 현재 코드와 대조한 뒤 이어서 완료한다.
7. `goal.md`가 없으면 아래 템플릿으로 새로 만든다.
8. 완료 조건과 테스트 계획이 구체적이지 않으면 구현을 시작하지 않는다.

### 2.1 `goal.md` 필수 템플릿

```md
# Goal

## 작업
- 한 문장으로 설명:
- 사용자에게 주는 가치:

## 범위
### 포함
- 

### 제외
- 

## 현재 상태
- 관련 화면/경로:
- 관련 테이블/마이그레이션:
- 관련 테스트:

## 가정과 결정 기록
- [날짜] 결정 또는 가정 / 이유 / 되돌릴 조건

## 완료 조건
- [ ] 사용자가 요청한 핵심 동작이 실제로 작동한다.
- [ ] 권한이 없는 사용자는 UI와 직접 요청 모두에서 차단된다.
- [ ] 로딩, 빈 상태, 오류 상태가 처리된다.
- [ ] 모바일 기준 화면에서 가로 스크롤 없이 사용할 수 있다.
- [ ] 필요한 테스트가 추가되고 통과한다.
- [ ] 문서와 마이그레이션이 현재 코드와 일치한다.

## 테스트 계획
- 정적 검사:
- 단위 테스트:
- 데이터베이스/RLS 테스트:
- E2E 테스트:
- 수동 확인 화면 크기:

## 위험과 되돌리기
- 위험:
- 롤백 방법:

## 검증 결과
- 실행한 명령:
- 결과:
- 남은 문제:
```

### 2.2 `goal.md` 종료 규칙

- 구현 후 체크박스를 추측으로 체크하지 않는다. 테스트 결과, 화면 확인, 쿼리 결과 등 근거가 있어야 한다.
- 하나라도 실패하면 `goal.md`를 유지하고 수정 → 재검증을 반복한다.
- 모든 완료 조건이 충족되고 관련 문서가 갱신된 경우에만 `goal.md`를 삭제한다.
- 테스트를 실행할 수 없는 환경이면 삭제하지 않는다. 실행하지 못한 이유와 필요한 다음 명령을 적는다.
- `goal.md`는 작업 중 안전장치이므로 `.gitignore`에 넣지 않는다. 미완료 파일이 남으면 작업이 끝나지 않았다는 신호다.

## 3. 제품의 북극성

이 제품의 핵심 사용자는 예배 직전이나 직후에 휴대폰 한 손으로 여러 아이를 빠르게 체크하는 바쁜 교사다.

모든 결정은 아래 질문으로 검증한다.

1. 교사가 휴대폰에서 빠르게 이해할 수 있는가?
2. 출석과 큐티 기록을 실수 없이 저장할 수 있는가?
3. 다른 반이나 다른 교회의 정보를 볼 수 없도록 막혀 있는가?
4. 관리자가 아이·반·교사를 쉽게 관리할 수 있는가?
5. 나중에 공개 서비스로 확장할 때 데이터 구조를 다시 만들지 않아도 되는가?

화려한 기능보다 **빠른 체크, 명확한 저장 상태, 안전한 권한**을 우선한다.

## 4. MVP 고정 범위

### 4.1 반드시 포함

- 로그인 없는 앱 진입
- 기본 반과 아이 목록
- 아이 추가
- 반별 주간 출석 및 큐티 체크
- 간단한 주간 메모
- Supabase 원격 저장과 새로고침 후 유지
- 이번 주 출석 인원 요약
- 이번 달 큐티 참여 아동 수와 완료 건수
- 이번 달 생일자 목록
- 모바일 우선 반응형 UI
- 기본 오류 처리, 테스트, 배포 문서

### 4.2 MVP에서 제외

- 로그인, 초대, 회원가입, 비밀번호, 역할 권한
- 중앙 공유 DB 저장, 조직·반 단위 RLS
- 공개 회원가입, 요금제, 결제
- 부모용 포털과 부모 알림
- 네이티브 iOS/Android 앱
- 완전한 오프라인 동기화
- 푸시 알림, 문자, 카카오톡 연동
- 사진·파일 첨부
- 복잡한 그래프와 예측 분석
- 여러 예배/부서의 복잡한 시간표
- CSV 대량 가져오기·내보내기
- 의료정보, 주민번호, 상세 가정사 등 불필요한 민감정보

제외 기능을 구현하려면 별도 작업 목표와 제품 책임자의 승인이 필요하다.

## 5. 기술 기준

### 5.1 기본 스택

- 프레임워크: 최신 안정 버전의 **Next.js App Router**
- 언어: **TypeScript**, `strict: true`
- 패키지 관리자: **pnpm**
- 스타일: **Tailwind CSS v4**와 **shadcn/ui**
- 저장: MVP는 **Supabase Postgres 정규화 업무 테이블**
- 향후 중앙 저장: **Supabase Postgres + Supabase Auth/RLS 또는 공유 코드 정책**
- 배포: **Vercel**
- 입력 검증: **Zod**
- 단위 테스트: **Vitest + React Testing Library**
- E2E: **Playwright**
- 아이콘: **Lucide**
- 토스트: shadcn/ui와 호환되는 **Sonner**

프로젝트를 시작할 때 최신 안정 호환 버전을 선택하고 lockfile에 고정한다. 한 작업 도중 주요 버전을 임의 업그레이드하지 않는다.

### 5.2 아키텍처 원칙

- Server Component를 기본으로 하고 실제 상호작용이 필요한 최소 영역만 Client Component로 만든다.
- 이번 MVP의 데이터 읽기와 쓰기는 Supabase 정규화 업무 테이블로 처리한다.
- 중앙 DB를 도입하면 데이터 읽기와 비즈니스 규칙은 `server-only` Data Access Layer를 통해 수행한다.
- 중앙 DB 쓰기는 Server Action 또는 필요한 Route Handler에서 수행하되, **모든 진입점에서 인증·권한 또는 공유 코드·입력값을 다시 검증**한다.
- Proxy/미들웨어는 편의용 라우팅 보호일 뿐 최종 권한 경계로 사용하지 않는다.
- 별도 Express/Nest 서버나 ORM은 MVP에 추가하지 않는다. Supabase typed client와 SQL migration으로 충분히 구현한다.
- 전역 상태 라이브러리는 사용하지 않는다. 서버 데이터와 화면 로컬 상태로 해결하고, 명확한 필요가 생긴 경우에만 도입한다.
- 출석 화면은 네트워크 호출을 행마다 발생시키지 않고 로컬에서 편집한 뒤 한 번에 저장한다.

### 5.3 권장 폴더 구조

```text

  (app)/
    dashboard/
    attendance/

  domain/             # 출석, 아이, 통계 등 도메인 컴포넌트
  layout/
```

기존 저장소가 다른 구조를 이미 확립했다면 무리한 전체 이동보다 기존 관례를 존중하되, 보안 경계와 도메인 분리는 유지한다.

## 6. 데이터베이스와 보안 규칙

이번 MVP는 로그인 없이 패밀리 오픈으로 사용하므로 업무 데이터는 Supabase 정규화 업무 테이블에 저장한다. 모든 핵심 테이블은 `organization_id`를 가지며, 현재는 기본 조직 1개에 대한 공개 읽기/쓰기를 허용한다. 공개 운영 전에는 Auth/RLS 또는 공유 코드 정책을 별도 `goal.md`로 다시 설계한다.

### 6.1 멀티테넌트 준비

중앙 DB를 도입하는 다음 단계에서는 한 교회만 사용하더라도 데이터 구조를 처음부터 조직 단위로 분리한다.

- 모든 핵심 테이블에 `organization_id`를 둔다.
- 쿼리와 mutation은 항상 현재 조직 범위를 명시한다.
- 다른 조직 ID를 URL이나 요청 본문에 넣어도 RLS가 차단해야 한다.
- 가능한 경우 조직 ID를 포함한 복합 외래키로 교차 조직 참조를 방지한다.

### 6.2 역할

- `owner`: 조직 소유자. 모든 설정과 관리자 관리 가능
- `admin`: 아이, 반, 교사 배정, 모든 반 기록과 통계 관리 가능
- `teacher`: 배정된 반의 아이 조회, 출석·큐티·특이사항 기록 가능

교사는 배정되지 않은 반의 아이와 기록을 직접 URL, Server Action, Supabase 요청 어느 방식으로도 볼 수 없어야 한다.

### 6.3 RLS

- 중앙 DB를 도입하면 공개 스키마의 모든 업무 테이블에 RLS 또는 동등한 공유 코드 제한 정책을 켠다.
- 중앙 DB 업무 테이블에 접근 제한이 없으면 완료로 간주하지 않는다.
- 브라우저에서 `secret`/service 역할 키를 사용하지 않는다.
- 관리자 초대처럼 권한 상승이 필요한 작업만 서버 전용 비밀키를 사용한다.
- RLS 정책은 정상 접근뿐 아니라 다른 조직·다른 반·권한 없는 쓰기 시나리오를 테스트한다.
- `SECURITY DEFINER` 함수는 꼭 필요한 경우에만 사용하고 `search_path`를 고정하며 별도 보안 검토를 한다.

### 6.4 환경변수

예상 이름은 다음과 같다.

```bash
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=
SUPABASE_SECRET_KEY=            # 서버 전용, 필요한 작업에만 사용
NEXT_PUBLIC_APP_URL=
```

- `.env*` 실제 값은 커밋하지 않는다.
- `.env.example`에는 이름과 설명만 둔다.
- `NEXT_PUBLIC_` 접두사는 브라우저에 노출되어도 되는 값에만 사용한다.
- 로그와 오류 메시지에 토큰, 초대 링크, 전화번호 등 민감정보를 남기지 않는다.

### 6.5 마이그레이션

- 스키마 변경은 반드시 `supabase/migrations`의 새 migration으로 남긴다.
- 운영 DB에서만 수동으로 만든 변경은 금지한다.
- 기존 migration을 배포 후 수정하지 않는다. 새 migration으로 교정한다.
- destructive migration에는 사전 백업·롤백·데이터 변환 검증을 `goal.md`에 적는다.
- 날짜/시간은 이벤트 시각은 `timestamptz` UTC, 예배일·생일 같은 달력 날짜는 `date` 또는 분리된 달/일 값으로 저장한다.

### 6.6 미성년자 개인정보 최소화

MVP 기본 아이 정보는 이름, 생일 월/일, 선택적 출생연도, 반 등록 상태, 내부 메모 정도로 제한한다.

- 보호자 연락처, 주소, 사진, 의료정보는 명시적 제품 결정 없이 추가하지 않는다.
- 내부 메모와 특이사항 UI에는 민감한 의료·가정 정보를 적지 말라는 안내를 둔다.
- 삭제보다 보관(archive)을 기본으로 하여 과거 출석 통계를 보존한다.
- 화면과 서버 응답은 필요한 필드만 반환한다.

## 7. 도메인 규칙

- 조직의 시간대는 필수 설정이다. 초기 설정에서 브라우저 시간대를 제안하되 관리자가 확정한다.
- 반별 예배 세션은 `class_id + session_date` 조합당 하나다.
- 한 세션에서 한 아이의 출석 기록은 하나만 존재한다.
- 세션은 `draft`와 `finalized` 상태를 가진다.
- 통계에는 기본적으로 `finalized` 세션만 포함한다.
- 출석 상태는 MVP에서 `present` 또는 `absent`다. 초안 상태에서는 미정 값을 허용할 수 있지만 완료 시 모두 확정되어야 한다.
- 큐티 체크는 출석과 독립적으로 저장한다.
- 월간 큐티 기본 지표는 다음 두 개를 함께 제공한다.
  - 참여 아동 수: 해당 월에 큐티 완료가 한 번 이상인 중복 제거 아동 수
  - 완료 건수: 해당 월의 모든 큐티 체크 합계
- 월간 생일자는 활성 아동 중 생일 월이 선택한 달과 같은 사람을 생일 일자, 반 순으로 보여준다.
- 아이가 반을 옮겨도 과거 세션은 당시 반에 남아야 한다.
- 동시에 두 교사가 같은 세션을 수정할 수 있으므로 세션 revision 또는 `updated_at` 기반 충돌 검사를 사용한다. 오래된 화면의 저장은 조용히 덮어쓰지 않고 새로고침 안내를 보여준다.

## 8. UI·UX 규칙

### 8.1 모바일 우선

- 360px 너비부터 설계한다.
- 주요 터치 영역은 최소 44×44px, 출석 토글과 저장 버튼은 48px 이상을 권장한다.
- 입력 폰트는 모바일 브라우저 확대를 피하도록 16px 이상 사용한다.
- 가로 스크롤을 만들지 않는다.
- 하단 고정 요소는 `safe-area-inset-bottom`을 반영한다.
- 핵심 출석 흐름은 대시보드에서 반 선택 후 바로 체크 화면에 도달해야 한다.
- 저장 전 변경 수와 저장 완료 시각을 명확히 보여준다.
- 네트워크 오류 시 입력을 즉시 버리지 말고 재시도할 수 있게 유지한다.

### 8.2 디자인 기준 적용

`DESIGN.md`의 밝고 친근한 Duolingo 계열 분위기를 참고하되 그대로 복제하지 않는다.

- 브랜드 핵심색: `#58cc02`
- 보조 상호작용색: `#1cb0f6`
- 카드·입력·버튼 radius: 12px
- 일반 카드에는 전통적인 그림자를 쓰지 않고 border와 여백으로 구분한다.
- 주요 버튼에만 단단한 하단 shadow를 사용해 눌리는 느낌을 준다.
- 교회·주일학교에 맞는 독창적인 캐릭터와 일러스트를 사용하고 Duolingo 상표, 부엉이, 고유 캐릭터를 모사하지 않는다.
- 일러스트는 대시보드 인사, 온보딩, 빈 상태처럼 의미가 있는 곳에 제한한다. 출석 명단의 정보 밀도를 방해하지 않는다.

### 8.3 접근성 보정

원본 색을 유지하면서 텍스트 대비를 확보한다.

- `#58cc02` 버튼에는 작은 흰색 글자 대신 `#3c3c3c` 또는 더 어두운 텍스트를 기본으로 사용한다.
- 원본 Sky Blue `#1cb0f6`는 아이콘·배경·강조에 쓰고, 흰 배경 링크 텍스트에는 더 어두운 파생색(예: `#0b79b7`)을 사용한다.
- 색만으로 출석 상태를 전달하지 않고 아이콘, 라벨, 체크 상태를 함께 제공한다.
- focus-visible outline을 제거하지 않는다.
- 스크린리더 라벨과 키보드 조작을 지원한다.

### 8.4 한글 타이포그래피

- 한글 헤드라인: `Jua` 계열 또는 라이선스가 명확한 둥근 한글 폰트
- 영문 헤드라인: `Fredoka`
- 한글 본문/UI: `Noto Sans KR`
- 영문·숫자 UI: `Nunito`
- `DESIGN.md`의 넓은 자간 `0.053em`은 영문 대문자 라벨에만 제한적으로 사용한다. 한글 본문에는 과도한 자간을 적용하지 않는다.
- 시스템 폰트만으로 끝내지 않되, 폰트 로딩 실패 시 읽을 수 있는 fallback을 둔다.

### 8.5 shadcn/ui와 Shadcnblocks

- 기본 UI는 shadcn/ui를 프로젝트 테마에 맞게 수정해 사용한다.
- Shadcnblocks는 완성도를 실제로 높이고 직접 만드는 비용을 줄일 때만 사용한다.
- 무료 블록도 소스와 접근성을 검토하고 프로젝트 토큰으로 재스타일링한다.
- Pro/Premium 블록이 필요하면 제품 책임자에게 API 키를 요청한다.
- 키는 `SHADCNBLOCKS_API_KEY` 환경변수로만 넣고 저장소에 커밋하지 않는다.
- 블록 도입 때문에 도메인 구조나 접근성을 희생하지 않는다.

## 9. 구현 품질 규칙

### 9.1 TypeScript

- `any` 사용을 피한다. 불가피하면 범위를 최소화하고 이유를 주석으로 남긴다.
- Supabase 타입은 스키마에서 생성하고 수동 중복 타입을 줄인다.
- 외부 입력은 타입 단언 전에 Zod로 검증한다.
- 날짜·상태·역할은 문자열을 흩뿌리지 않고 공통 타입과 상수로 관리한다.

### 9.2 컴포넌트

- 화면 파일에 거대한 폼과 도메인 로직을 모두 넣지 않는다.
- 재사용 가능한 UI와 도메인 컴포넌트를 분리한다.
- 과도하게 일반화된 추상 컴포넌트를 먼저 만들지 않는다. 두 번 이상 실제 반복될 때 추출한다.
- Client Component 경계를 작게 유지하고 서버 전용 객체를 props로 넘기지 않는다.

### 9.3 서버 로직

중앙 DB를 도입한 뒤 모든 mutation은 아래 순서를 따른다.

1. 로그인 사용자 또는 공유 코드 확인
2. 조직 멤버십·역할 또는 공유 범위 확인
3. 대상 반 접근 권한 또는 공유 범위 확인
4. Zod 입력 검증
5. 조직 범위를 포함한 데이터 변경
6. 최소한의 결과만 반환
7. 필요한 경로 재검증 또는 UI 상태 갱신

이번 로그인 없는 Supabase MVP에서는 별도 Server Action mutation이 없다. Server Action을 도입하면 UI에서만 호출된다고 가정하지 않는다. 직접 POST 요청이 가능한 공개 진입점처럼 다룬다.

### 9.4 오류와 상태

모든 주요 화면은 아래 상태를 갖는다.

- loading/skeleton
- empty state와 다음 행동
- validation error
- 권한 없음
- 네트워크/서버 오류와 재시도
- 성공 피드백
- 저장 중 중복 제출 방지

사용자 메시지는 “왜 실패했는지”와 “다음에 무엇을 할지”를 짧게 알려야 한다. 데이터베이스 원문 오류를 그대로 노출하지 않는다.

### 9.5 성능

- 출석 목록은 한 번의 쿼리 또는 소수의 예측 가능한 쿼리로 가져온다.
- N+1 쿼리를 피한다.
- 통계는 필요한 열만 선택하고 조직·날짜·반 인덱스를 사용한다.
- 인증 화면과 내부 화면에 불필요한 대형 이미지를 로드하지 않는다.
- MVP 규모에서 측정 없이 복잡한 캐시를 추가하지 않는다.

## 10. 테스트 기준

### 10.1 필수 명령

프로젝트 `package.json`에는 최소한 아래 의미의 스크립트를 둔다.

```bash
pnpm lint
pnpm typecheck
pnpm test
pnpm test:db
pnpm test:e2e
pnpm build
```

실제 도구 명령은 스크립트 뒤에 캡슐화하여 개발자와 Codex가 같은 명령을 사용하게 한다.

### 10.2 테스트 범위

- 단위 테스트: 날짜 계산, 주간/월간 집계, 권한 판정 보조 함수, Zod schema
- DB/RLS 테스트: 이번 MVP에서는 정규화 업무 테이블, 기본 RLS, 기존 JSON 이관 SQL이 migration에 포함되어 있는지 확인한다. 권한 모델 도입 시 같은 조직 정상 접근, 다른 조직 차단, 다른 반 교사 차단, 관리자 허용을 테스트한다.
- E2E: 로그인 없이 진입, 아이 추가, 출석·큐티 저장, 메모, 통계, 생일자
- 회귀 테스트: 같은 세션 중복 생성 방지, 동일 아동 중복 기록 방지, 오래된 revision 저장 거절
- 반응형 확인: 최소 390×844 모바일과 1280px 데스크톱
- 브라우저: Chromium과 WebKit을 최소 기준으로 하며 가능하면 Firefox도 실행한다.

### 10.3 접근성 확인

- 키보드만으로 주요 폼 제출 가능
- 각 토글의 이름과 현재 상태가 스크린리더에 전달됨
- 오류가 입력과 연결됨
- 텍스트 대비와 focus 상태 확인
- 모달·드로어 열림 시 focus 관리

## 11. 완료 정의(Definition of Done)

다음이 모두 참이어야 작업이 완료다.

- `goal.md`의 사용자 가치와 범위가 구현되었다.
- 완료 조건이 실제 증거와 함께 체크되었다.
- lint, typecheck, 관련 단위/DB/E2E 테스트가 통과했다.
- production build가 성공했다.
- 모바일과 데스크톱에서 핵심 흐름을 확인했다.
- 중앙 DB 또는 서버 mutation을 도입한 경우 권한 없는 직접 접근이 서버와 RLS 또는 공유 코드 정책에서 차단된다.
- 로딩, 빈 상태, 오류, 성공 상태가 처리됐다.
- 새 스키마 변경이 있으면 migration에 있고 재현 가능하다.
- 새 환경변수가 `.env.example`과 배포 문서에 반영됐다.
- 콘솔 오류와 미해결 경고가 없다.
- 관련 문서가 실제 동작과 일치한다.
- 남은 위험이 있으면 사용자에게 명확히 보고됐다.

모두 충족하면 마지막으로 `goal.md`를 삭제한다.

## 12. 금지 사항

- 테스트를 실행하지 않고 “완료”라고 보고하기
- RLS 없이 UI 조건문만으로 권한을 막기

- 운영 데이터를 직접 수정하고 migration을 생략하기
- 아이 기록을 기본적으로 hard delete 하기
- 요청 범위와 무관한 대규모 리팩터링 수행하기
- 데이터 손실 가능성이 있는 작업을 조용히 실행하기
- 디자인을 이유로 접근성 대비와 터치 크기를 희생하기
- Duolingo의 고유 캐릭터나 상표를 복제하기
- Pro Shadcnblocks API 키를 코드, 로그, 커밋에 남기기
