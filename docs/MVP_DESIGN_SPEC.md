## 1. 제품 개요

### 1.1 한 문장 설명

가족과 가까운 교회 구성원이 계정 없이 바로 열어 반 아이들의 주간 출석과 큐티 여부를 빠르게 기록하고, 간단한 생일·출석·큐티 현황을 확인하는 모바일 우선 웹앱이다.

### 1.2 이번 MVP의 성격

이번 MVP는 공개 SaaS가 아니라 **패밀리 오픈**이다.

- 로그인, 초대, 역할 권한, 결제는 만들지 않는다.
- 한 교회 또는 한 가족 규모의 사용을 전제로 한다.
- Supabase Postgres의 단일 JSON 상태 행을 사용해 여러 기기에서 같은 데이터를 공유한다.
- 로그인 없는 패밀리 오픈이므로 publishable key 기반 공개 읽기/쓰기를 허용한다.
- `localStorage`는 원격 저장 실패 시 임시 백업과 초기 마이그레이션 용도로만 사용한다.

### 1.3 성공 기준

- 휴대폰에서 3단계 이내로 출석 체크 화면에 도달한다.
- 아이별 출석과 큐티를 한 손으로 빠르게 바꿀 수 있다.
- 저장 여부가 명확하고 새로고침·기기 변경 후에도 Supabase에 저장된 데이터가 유지된다.
- 아이 상세 정보와 보호자 연락처는 Supabase 원격 상태 JSON에 저장된다.
- 모바일 390px 화면에서 가로 스크롤 없이 사용할 수 있다.

1. 로그인 없는 홈 진입

4. 반별 출석/결석 체크

18. 홈과 출석 반 선택에 담임 선생님 표시
19. 전도사님 공유 메모 체크
20. lint/typecheck/unit/E2E/build 검증

### 2.2 제외

- 로그인, 초대, 비밀번호, 회원가입
- 역할 기반 권한, 교사 배정 권한, 관리자 권한
- Supabase Auth, 정규화된 업무 테이블, 조직/반 단위 RLS
- 여러 조직/교회 전환
- 서버 사진 업로드와 외부 파일 저장
- CSV 가져오기/내보내기
- 결제, 공개 SaaS 운영 기능

### 2.3 나중에 결정할 사항

- 공개 쓰기 상태를 언제 공유 코드 또는 Supabase Auth로 막을 것인가?
- 생일 연도가 필요한가, 월/일만으로 충분한가?
- 반이 늘어날 때 반 생성/수정 UI가 필요한가?
- 출석 상태가 `present/absent` 외에 지각, 공결까지 필요한가?

## 4. 핵심 흐름

### 4.1 출석 체크

1. 앱을 연다.
2. 홈 또는 하단 탭에서 출석을 누른다.
3. 아이별 출석과 큐티 완료를 체크한다.
4. 필요하면 이번 주 메모를 적는다.
5. 저장 버튼을 눌러 저장한다.
6. 저장 상태를 확인한다.

### 4.2 아이 추가

1. 아이들 탭을 연다.
2. 아이 추가 버튼을 누른다.
3. 모달에서 이름, 사진, 성별, 생년월일, 보호자, 주소, 이메일, 등록일, 반, 특이사항을 입력한다.
4. 저장하면 목록과 출석 화면에 반영된다.

### 4.3 통계 확인

1. 통계 탭을 연다.
2. 이번 주 출석 수, 이번 달 큐티 참여/완료 건수, 생일자를 확인한다.
3. 주간 출석, 월간 큐티, 월간 생일 카드를 눌러 상세 명단을 확인하고 복사한다.

## 5. 데이터 모델

### 5.1 Supabase 저장 모델

Supabase `family_open_app_state` 테이블의 `default` 행에 하나의 JSON 상태를 저장한다. `localStorage`는 초기 로딩 백업과 원격 오류 fallback으로만 사용한다.

```ts
type FamilyOpenStore = {
  version: 1
  teachers: Array<{
    id: string
    name: string
    photoDataUrl?: string
    birthDate?: string
    birthMonth?: number
    birthDay?: number
    phone?: string
    isActive: boolean
  }>
  classes: Array<{ id: string; name: string; teacherId?: string }>
  children: Array<{
    id: string
    name: string
    classId: string
    photoDataUrl?: string
    gender?: "male" | "female" | "unspecified"
    birthDate?: string
    birthYear?: number
    birthMonth: number
    birthDay: number
    parents?: Array<{ id: string; name: string; phone: string }>
    address?: string
    email?: string
    registeredAt?: string
    notes?: string
    isActive: boolean
  }>
  attendanceByDate: Record<
    string,
    {
      sessionDate: string
      records: Record<string, { status?: "present" | "absent"; qtCompleted: boolean }>
      note: string
      shareWithPastor?: boolean
      savedAt: string
    }
  >
}
```

### 5.2 개인정보 원칙

- 사용자가 명시적으로 요청한 아이 상세 정보와 보호자 연락처를 입력할 수 있다.
- 사진은 서버 파일 업로드 없이 브라우저에서 Data URL로 변환해 Supabase 상태 JSON에 저장한다.
- 사진은 500KB 이하로 제한한다.
- 특이사항은 알러지 등 출석 관리에 필요한 짧은 내용만 적는 것을 전제로 한다.
- Supabase migration이 적용되지 않았거나 네트워크 오류가 있으면 저장 실패 상태를 표시한다.

### 5.3 향후 보안 전환

공개 배포 또는 외부 사용자 접근 전에 새 `goal.md`로 다음을 다시 설계한다.

- 로그인 또는 공유 코드 방식
- 업무 테이블 migration
- RLS 또는 공유 키 기반 제한 정책
- 정규화된 업무 테이블과 백업/복구
- 다중 기기 충돌 처리

## 7. UI·UX 기준

- 360px 너비부터 설계한다.
- 주요 터치 영역은 최소 44×44px, 출석 토글은 48px 이상을 권장한다.
- 입력 폰트는 16px 이상 사용한다.
- 하단 내비게이션은 safe area를 반영한다.
- 가로 스크롤을 만들지 않는다.
- 저장 상태는 "저장됨", "저장 중", "저장 실패"처럼 명확히 보여준다.
- `DESIGN.md`의 밝고 친근한 분위기, `#58cc02` CTA, `#1cb0f6` 보조색, 12px radius를 사용한다.
- 색만으로 상태를 전달하지 않고 텍스트와 아이콘을 함께 사용한다.

## 11. 2026-06-26 관리 기능 보강

- `/teachers`는 등록된 선생님 카드에서 수정 진입을 제공하고, 삭제는 수정 모달 안에서 제공한다.
- 선생님 등록/수정 모달의 생일 입력은 연도 없이 월/일만 받는다.
- 선생님 삭제는 `isActive: false`로 숨기고, 해당 선생님이 담임인 반의 `teacherId`를 비운다.
- `/settings`는 등록된 반 카드에서 수정 진입을 제공하고, 삭제는 수정 모달 안에서 제공한다.
- 반 삭제는 반을 제거하고 해당 반 아이들을 반 미지정으로 전환한다.
- 공통 저장 상태는 저장 시각 없이 `저장됨`, `저장 중`, `저장 실패`만 표시한다.
- E2E는 Supabase 상태 초기화, 선생님 수정/삭제, 반 수정/삭제, 원격 저장 후 새로고침 유지를 검증한다.

## 12. 2026-06-26 미지정 등록 허용

- 아이 등록/수정 모달은 반을 선택하지 않아도 저장할 수 있다.
- 반 미지정 아이는 아이들 화면의 전체 목록에는 표시되고, 특정 반 출석 화면에는 자동 포함되지 않는다.
- 선생님 등록/수정 모달은 반을 선택하지 않아도 저장할 수 있다.
- 반 미지정 선생님은 어떤 반의 담임도 차지하지 않는다.
- 반 등록/수정은 담임 선생님을 선택하지 않아도 저장할 수 있다.
- 담임 미지정 반은 홈/출석/설정 화면에서 미지정 상태로 표시한다.
- Supabase 단일 JSON 저장 방식은 유지하며 별도 migration은 추가하지 않는다.

## 13. 2026-06-26 삭제 confirm 및 모달 위치

- 아이, 선생님, 반 삭제는 목록 카드에서 직접 실행하지 않는다.
- 삭제 버튼은 해당 항목의 상세/수정 모달 안에 위치한다.
- 삭제 전에는 브라우저 기본 `window.confirm`으로 사용자 확인을 받는다.
- 아이 삭제는 출석 기록 참조 보존을 위해 `isActive: false`로 숨긴다.
- 선생님 삭제는 `isActive: false`로 숨기고, 해당 선생님이 담임인 반의 `teacherId`를 비운다.
- 반 삭제는 반을 제거하고, 그 반에 연결된 아이는 `classId: ""`로 변경해 반 미지정 상태로 남긴다.
- 모달 삭제 버튼은 저장/취소 액션과 분리된 위험 액션 스타일을 사용한다.
