- 문서 버전: 2.0
- 최초 작성일: 2026-06-23
- 범위 변경일: 2026-06-25
- 상태: 로그인 없는 패밀리 오픈 MVP 기준선
- 연관 문서: `/instruction.md`, `/DESIGN.md`, `/AGENTS.md`, `/docs/PROGRESS.md`

## 1. 제품 개요

### 1.1 한 문장 설명

가족과 가까운 교회 구성원이 계정 없이 바로 열어 반 아이들의 주간 출석과 큐티 여부를 빠르게 기록하고, 간단한 생일·출석·큐티 현황을 확인하는 모바일 우선 웹앱이다.

### 1.2 이번 MVP의 성격

이번 MVP는 공개 SaaS가 아니라 **패밀리 오픈**이다.

- 로그인, 초대, 역할 권한, 결제는 만들지 않는다.
- 한 교회 또는 한 가족 규모의 사용을 전제로 한다.
- 중앙 DB를 공개 쓰기로 열지 않는다.
- 첫 구현은 브라우저 `localStorage` 저장을 사용한다.
- 여러 기기 공유가 필요해지면 별도 제품 결정 후 Supabase 저장 방식을 설계한다.

### 1.3 성공 기준

- 휴대폰에서 3단계 이내로 출석 체크 화면에 도달한다.
- 아이별 출석과 큐티를 한 손으로 빠르게 바꿀 수 있다.
- 저장 여부가 명확하고 새로고침 후에도 같은 브라우저에서는 유지된다.
- 아이 정보는 이름, 생일 월/일, 반 정도로 제한한다.
- 모바일 390px 화면에서 가로 스크롤 없이 사용할 수 있다.

## 2. MVP 범위

### 2.1 Must Have

1. 로그인 없는 홈 진입
2. 기본 반과 아이 목록
3. 아이 추가
4. 반별 출석/결석 체크
5. 아이별 큐티 완료 체크
6. 간단한 주간 메모
7. 브라우저 로컬 저장과 새로고침 후 유지
8. 이번 주 출석 요약
9. 이번 달 큐티 참여 아동 수와 완료 건수
10. 이번 달 생일자 목록
11. 모바일 우선 하단 내비게이션
12. 기본 빈 상태, 저장 상태, 오류 방어
13. lint/typecheck/unit/E2E/build 검증

### 2.2 제외

- 로그인, 초대, 비밀번호, 회원가입
- 역할 기반 권한, 교사 배정 권한, 관리자 권한
- Supabase 업무 테이블, RLS, 원격 DB 저장
- 여러 조직/교회 전환
- 보호자 연락처, 주소, 사진, 의료정보
- CSV 가져오기/내보내기
- 결제, 공개 SaaS 운영 기능

### 2.3 나중에 결정할 사항

- 여러 기기에서 같은 데이터를 공유해야 하는가?
- 공유가 필요하다면 공개 편집 링크, 간단한 가족 코드, Supabase Auth 중 무엇을 쓸 것인가?
- 생일 연도가 필요한가, 월/일만으로 충분한가?
- 반이 늘어날 때 반 생성/수정 UI가 필요한가?
- 출석 상태가 `present/absent` 외에 지각, 공결까지 필요한가?

## 3. 정보 구조와 라우트

```text
/
/dashboard
/attendance
/children
/reports
```

- `/`는 `/dashboard`로 이동한다.
- `/login`, `/auth/callback`, `/invite/accept`는 MVP에서 사용하지 않는다.
- 하단 탭은 홈, 출석, 아이들, 통계 4개로 유지한다.

---

## 4. 핵심 흐름

### 4.1 출석 체크

1. 앱을 연다.
2. 홈 또는 하단 탭에서 출석을 누른다.
3. 아이별 출석/결석과 큐티 완료를 체크한다.
4. 필요하면 이번 주 메모를 적는다.
5. 저장 상태를 확인한다.

### 4.2 아이 추가

1. 아이들 탭을 연다.
2. 이름, 생일 월/일, 반을 입력한다.
3. 저장하면 목록과 출석 화면에 반영된다.

### 4.3 통계 확인

1. 통계 탭을 연다.
2. 이번 주 출석 수, 이번 달 큐티 참여/완료 건수, 생일자를 확인한다.

## 5. 데이터 모델

### 5.1 브라우저 저장 모델

`localStorage`에 하나의 JSON 상태를 저장한다.

```ts
type FamilyOpenStore = {
  version: 1
  classes: Array<{ id: string; name: string }>
  children: Array<{
    id: string
    name: string
    classId: string
    birthMonth: number
    birthDay: number
    isActive: boolean
  }>
  attendanceByDate: Record<
    string,
    {
      sessionDate: string
      records: Record<string, { status: "present" | "absent"; qtCompleted: boolean }>
      note: string
      savedAt: string
    }
  >
}
```

### 5.2 개인정보 원칙

- 이름은 별명 또는 짧은 이름을 권장한다.
- 생일은 월/일만 저장한다.
- 보호자 연락처, 사진, 의료/가정 정보는 저장하지 않는다.
- 브라우저 저장소를 지우면 데이터가 사라질 수 있음을 UI와 문서에 알린다.

### 5.3 향후 중앙 DB 전환

중앙 저장이 필요해지면 Supabase Postgres를 사용한다. 그 시점에는 새 `goal.md`로 다음을 다시 설계한다.

- 로그인 또는 공유 코드 방식
- 업무 테이블 migration
- RLS 또는 공유 키 기반 제한 정책
- 백업/복구
- 다중 기기 충돌 처리

## 6. 화면 설계

### 6.1 홈 `/dashboard`

- 오늘 날짜
- 출석 체크 시작 버튼
- 이번 주 출석 요약
- 이번 달 큐티 요약
- 이번 달 생일 요약
- 로컬 저장 안내

### 6.2 출석 `/attendance`

- 날짜 선택
- 반 선택
- 전체 출석 버튼
- 아이별 출석/결석 토글
- 아이별 큐티 완료 토글
- 이번 주 메모
- 저장 상태

### 6.3 아이들 `/children`

- 아이 목록
- 아이 추가 폼
- 빈 상태
- 아이별 이름, 반, 생일 표시

### 6.4 통계 `/reports`

- 이번 주 출석 인원
- 이번 달 큐티 참여 아동 수
- 이번 달 큐티 완료 건수
- 이번 달 생일자 목록

## 7. UI·UX 기준

- 360px 너비부터 설계한다.
- 주요 터치 영역은 최소 44×44px, 출석 토글은 48px 이상을 권장한다.
- 입력 폰트는 16px 이상 사용한다.
- 하단 내비게이션은 safe area를 반영한다.
- 가로 스크롤을 만들지 않는다.
- 저장 상태는 "저장됨", "저장 중", "저장 실패"처럼 명확히 보여준다.
- `DESIGN.md`의 밝고 친근한 분위기, `#58cc02` CTA, `#1cb0f6` 보조색, 12px radius를 사용한다.
- 색만으로 상태를 전달하지 않고 텍스트와 아이콘을 함께 사용한다.

## 9. 테스트 전략

### 9.1 단위 테스트

- 날짜 계산
- 생일 월/일 검증
- 출석/큐티 통계 계산
- 로컬 저장 reducer 또는 helper

### 9.2 DB/RLS 테스트

이번 MVP는 중앙 DB를 사용하지 않는다. `test:db`는 "업무 migration 없음"과 "no-login/local-first 범위"를 확인하는 기준선 테스트로 둔다.

### 9.3 E2E

- 로그인 없이 `/dashboard` 진입
- 하단 탭 이동
- 아이 추가
- 출석/큐티 체크
- 새로고침 후 유지
- 통계 반영
- 390px 모바일과 1280px 데스크톱에서 가로 스크롤 없음
- Chromium과 WebKit 확인

## 10. 완료 정의

- 문서가 no-login 패밀리 오픈 범위와 일치한다.
- 로그인 없이 핵심 화면을 사용할 수 있다.
- 아이 추가와 출석/큐티 체크가 같은 브라우저에서 유지된다.
- 민감정보 입력을 요구하지 않는다.
- lint/typecheck/unit/DB/E2E/build가 통과한다.
- 진행 기록이 `docs/PROGRESS.md`에 남는다.
